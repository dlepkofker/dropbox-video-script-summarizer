# Codebase Structure

**Analysis Date:** 2026-04-11

## Directory Layout

```
dropbox-video-script-summarizer/
├── src/                  # Frontend SPA source (compiled by Vite)
│   ├── assets/           # Static assets imported by TypeScript modules
│   ├── main.ts           # UI orchestration — all views and event handling
│   ├── dropbox.ts        # Dropbox SDK wrapper (video listing, temp links)
│   ├── assemblyai.ts     # AssemblyAI HTTP client (submit + poll)
│   ├── supabase.ts       # Server REST client for all Supabase-backed resources
│   └── style.css         # Global stylesheet for all views
├── server/               # Express backend (separate Node.js process / package)
│   ├── index.ts          # Entire server: auth, proxies, CRUD, AI generation
│   ├── package.json      # Server-only dependencies (express, openai, etc.)
│   ├── tsconfig.json     # Server TypeScript config (ESNext, Node types)
│   ├── .env.example      # Required server env vars template
│   └── node_modules/     # Server-only packages (not shared with frontend)
├── public/               # Static files served by Vite as-is (no processing)
│   ├── favicon.svg
│   └── icons.svg
├── index.html            # SPA shell — nav + #app mount point
├── package.json          # Frontend dependencies (dropbox, marked, dompurify)
├── tsconfig.json         # Frontend TypeScript config
├── .prettierrc           # Shared Prettier formatting config
├── .env.example          # Frontend env vars template (VITE_SERVER_URL)
├── Dockerfile            # Container image for production deployment
├── fly.toml              # Fly.io deployment config
└── .planning/            # GSD planning artefacts (not committed to main)
    └── codebase/         # Codebase analysis documents
```

## Directory Purposes

**`src/`:**
- Purpose: All browser-executed code
- Contains: TypeScript modules compiled and bundled by Vite; one CSS file
- Key files: `src/main.ts` (UI), `src/dropbox.ts`, `src/assemblyai.ts`, `src/supabase.ts`
- Build output: `dist/` (gitignored, generated by `npm run build`)

**`server/`:**
- Purpose: Express REST backend — secrets proxy, ffmpeg pipeline, Supabase CRUD, OpenAI generation
- Contains: Single TypeScript source file plus its own `package.json` and lockfile
- Key files: `server/index.ts`
- Run with: `tsx index.ts` (dev) or compiled output (production)
- Has its own `node_modules/` — entirely separate dependency tree from the frontend

**`public/`:**
- Purpose: Static assets copied verbatim into the Vite build output
- Contains: SVG icons referenced from `index.html`
- Generated: No — these are source files
- Committed: Yes

**`dist/`** (not present in source):
- Purpose: Vite production build output
- Generated: Yes, via `npm run build`
- Committed: No (gitignored)
- Served by: Express `express.static(distPath)` in production

## Key File Locations

**Entry Points:**
- `index.html` — browser entry point; loads `src/main.ts` as ES module
- `server/index.ts` — server entry point; single `app.listen()` call at EOF

**Configuration:**
- `tsconfig.json` — frontend TypeScript (targets ESNext, bundler module resolution)
- `server/tsconfig.json` — server TypeScript (targets ESNext, NodeNext resolution)
- `.prettierrc` — formatting rules shared across both `src/` and `server/`
- `fly.toml` — Fly.io app config (port, region, health checks)
- `Dockerfile` — multi-stage build: Node install → `npm run build` → server start

**Core Logic:**
- `src/main.ts` — all UI state, view rendering functions, event listeners (~780 lines)
- `server/index.ts` — all route handlers, token management, ffmpeg orchestration (~579 lines)
- `src/dropbox.ts` — Dropbox SDK usage, `VideoFile` type definition
- `src/supabase.ts` — `Prompt` and `Instruction` type exports; all server REST wrappers

**Environment Templates:**
- `.env.example` — frontend vars (`VITE_SERVER_URL`, `VITE_DROPBOX_ACCESS_TOKEN`)
- `server/.env.example` — server vars (`DROPBOX_APP_KEY`, `DROPBOX_APP_SECRET`, `ASSEMBLYAI_API_KEY`, `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `OPENAI_API_KEY`)

## Naming Conventions

**Files:**
- `kebab-case` for config files (`.env.example`, `fly.toml`)
- `camelCase` for TypeScript source files (`main.ts`, `dropbox.ts`, `assemblyai.ts`, `supabase.ts`)
- Single-word or compound-word CSS file (`style.css`)

**Functions:**
- `camelCase` throughout: `listAllVideos`, `getTemporaryLink`, `renderVideoList`, `fetchDropboxToken`
- Render functions prefixed with `render`: `renderConnectView`, `renderLoading`, `renderError`

**Types/Interfaces:**
- `PascalCase`: `VideoFile`, `Prompt`, `Instruction`, `Tokens`, `TranscriptResponse`

## File Organization Principles

- **Frontend modules are thin clients:** `dropbox.ts`, `assemblyai.ts`, and `supabase.ts` each own one domain and export only what `main.ts` needs. No cross-imports between them.
- **Server is a single file:** All Express routes, middleware, token management, and external SDK calls live in `server/index.ts`. Sections are delimited by banner comments (e.g., `// ── Dropbox auth ──`).
- **Frontend and server have independent `package.json` files:** Running `npm install` in the root installs frontend deps; running it inside `server/` installs server deps. They share no `node_modules`.

## Where to Add New Code

**New frontend view or UI section:**
- Add a `renderXxx(): string` pure function in `src/main.ts`
- Wire event listeners in the same file after calling `innerHTML =`
- Add styles to `src/style.css`

**New external API integration (frontend-facing):**
- Create `src/{service}.ts` following the pattern of `src/assemblyai.ts` — export async functions, use `fetch` against server proxy routes, no SDK imports
- Add corresponding proxy route(s) in `server/index.ts`

**New server route / resource:**
- Add route handlers to `server/index.ts` within an appropriately labelled section (`// ── Resource Name ──`)
- If the resource needs Supabase, add CRUD operations following the existing `prompts` or `instructions` pattern

**New Supabase-backed resource (client types):**
- Add `export interface ResourceName { id: number; ... }` and CRUD functions to `src/supabase.ts`

**Static assets:**
- SVG/image files referenced from HTML: place in `public/`
- Assets imported from TypeScript: place in `src/assets/`

## Special Directories

**`.planning/`:**
- Purpose: GSD planning documents and codebase analysis
- Generated: Partially (by GSD map/plan commands)
- Committed: No (not in main; present on feature branches)

**`server/node_modules/`:**
- Purpose: Server-only runtime dependencies
- Generated: Yes, via `npm install` inside `server/`
- Committed: No
